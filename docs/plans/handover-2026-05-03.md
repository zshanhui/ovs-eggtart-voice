## 2. ASR 修复（已完成）

### 根因

`trt_edge_llm_ipc.py` 的 `audio_bytes_to_mel` 用 scipy 实现 mel spectrogram，与模型训练时使用的 librosa 不兼容。

**具体差异**：
- scipy 版：固定 `-4dB` Whisper clamp → `np.clip(log_mel, -4.0, None)`
- librosa 版（旧工作 runtime）：动态 `max-8dB` clamp → `np.maximum(log_spec, log_spec.max() - 8.0)`

**修复**：`app/backends/trt_edge_llm_ipc.py` 的 `audio_bytes_to_mel` 改为 librosa STFT + Slaney mel filterbank。

### 官方 PR 方向

当前可用实现仍带两个临时点：专用 CP engine 和 fp32 embedding 通过环境变量指定，并且启动时仍加载了官方通用 CP runner。给官方提 PR 时建议拆成两步：

1. **修 Talker/TTS 语义问题**
   - raw text token 序列使用 `[151644, 77091, 198] + tokenizer.encode(rawText)`。
   - prefill 使用 9-row 非 streaming layout。
   - `small_to_mtp_projection` 在 hidden size 相同时必须 identity/copy。
   - residual 只累加 active CP groups，inactive codebook zero-fill。
2. **新增 Qwen3-TTS 专用 CP runner**
   - 不把 Qwen3-TTS CP 当普通 LLM runner。
   - 支持 `inputs_embeds/cache_position/past_length/gen_step/past_key/past_value/new_past_key/new_past_value` engine 接口。
   - CP engine/embedding 从 `code_predictor/config.json` 或 engine tensor names 自动识别，不依赖环境变量。
   - 专用 CP runner 启用时跳过通用 CP engine loading，降低显存占用。

根因判断：官方 EdgeLLM 0.7.0 的 Qwen3-TTS CP 适配把 CP 走成通用 LLM decode，缺少 Qwen3-TTS 专用 `cache_position/past_length/gen_step` 语义，导致 residual code drift，最终语义错误。

### 8GB Nano/NX 内存结论

完整同驻不可行：

- 原 ASR thinker + TTS worker：TTS worker 启动阶段退出。
- ASR `thinker_kv512` + TTS worker + lazy Code2Wav：可启动，但 `MemAvailable` 约 716MB；ASR 后约 511MB，无法安全直接 Code2Wav。
- ASR `thinker_pruned35k_kv512` + TTS worker + lazy Code2Wav：完整同驻可运行，但 Code2Wav 会明显换页，TTS 变慢。
- 释放 ASR worker 后再 TTS 可运行：

```text
ASR_PRELOAD 7.5s
TTS_PRELOAD 11.9s
ASR 4.36s (含 Python 首次 mel 3.37s)
TTS 3.79s (lazy Code2Wav 首次加载约 2.05s)
V2V total 8.47s
```

这个结果说明：当前 8GB 设备上，Qwen3-ASR EdgeLLM thinker/audio encoder 与 Qwen3-TTS talker/CP/code2wav 不能作为完整双常驻服务跑到 <1s V2V。要继续压延迟，需要至少一项成立：

1. 16GB Orin NX/AGX 上验证完整双常驻。
2. ASR thinker 做更深内存优化（小 vocab embedding/pruned thinker、低精度/权重共享、或更小模型）。
3. TTS Code2Wav 做 chunk/window streaming，首包不等完整 wav。
4. 对话编排改成单活 GPU worker：ASR active during listening，TTS active during speaking；这可运行但不满足亚秒 V2V。

## 12. 2026-05-03 输入侧 Vocab Pruning 适配结果

旧版 `token_map.bin` 的输入侧 indirection 已迁到 EdgeLLM runtime 的 vanilla base-model 路径：

- EdgeLLM 加载官方 `vocab_map.safetensors` 后构建 host 端 `orig_id -> reduced_id`。
- 当 `embedding.safetensors` 行数等于 `reduced_vocab_size` 时，prefill/decoding 查 embedding 前把普通文本 token 映射到 reduced ID。
- audio/image special token 保持完整 ID，用于 Qwen3-Omni multimodal embedding kernel 识别；采样输出仍用官方 `reduced -> full` map 回完整 ID，tokenizer decode 不变。
- 当前只支持 vanilla base model，不支持 Eagle draft；legacy `tokenId >= vocabSize` vision insertion 遇到 reduced embedding 会直接报错，避免错误索引。

结论：旧 vocab pruning 的输入侧适配可以复用到官方 EdgeLLM，但由于官方 pruned35k 版本已经把 full embedding 做成 FP8，35k FP16 embedding 只从 154MB 降到约 73MB，实际 `MemAvailable` 改善不明显。若继续压内存，需要把 reduced embedding 也做 FP8，理论上可降到约 35-40MB。
