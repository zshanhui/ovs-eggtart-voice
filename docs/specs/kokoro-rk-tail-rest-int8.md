# Kokoro RK — Tail-rest CPU ONNX INT8 dynamic quantization (P7a)

**Status: SHIPPED 2026-05-23 (env-gated, opt-in default, performance neutral).**
ORT dynamic INT8 quantization applied to the 3 bucket-specific `kokoro-vocoder-tail-rest-cpu*.onnx`
files (bucket-8/16/32). Code path env-gated, container-deployed and
production-stable, but **measured TTFA improvement is within run-to-run
noise (±5 %), not the spec-projected 15–20 %.** The work is recorded here for
reproducibility and as a definitive negative-result note for the
"INT8 the CPU tail" branch on the bucket-32 4-stage pipeline.

## 1. Scope

Three artifacts, all CPU-side ORT InferenceSessions of the 4-stage
`vocoder-front-half (RKNN FP16) → tail-rest (CPU ONNX)` split:

| Bucket | FP32 file | INT8 file | FP32 size | INT8 size | Ratio |
|---|---|---|---|---|---|
| 8 | `kokoro-vocoder-tail-rest-cpu-bucket8.onnx` | `…-bucket8.int8.onnx` | 18.74 MB | 16.52 MB | 0.88 |
| 16 | `kokoro-vocoder-tail-rest-cpu-bucket16.onnx` | `…-bucket16.int8.onnx` | 21.52 MB | 19.30 MB | 0.90 |
| 32 | `kokoro-vocoder-tail-rest-cpu.onnx` | `…cpu.int8.onnx` | 26.39 MB | 24.18 MB | 0.92 |

Compression ratio averages **~0.90**: indicates only ~30–35 % of the tail-rest
parameter footprint is in MatMul/Gemm ops (which we quantize); the rest are
Conv1D / activation buffers that we deliberately left untouched (Conv
quantization on the Kokoro postnet shifted activation distributions in
exploratory tests and degraded audio rel_l2).

## 2. Quantization recipe

`onnxruntime.quantization.quantize_dynamic` with:

```python
quantize_dynamic(
    src, dst,
    op_types_to_quantize=["MatMul", "Gemm"],
    weight_type=QuantType.QInt8,
)
```

- ORT 1.26.0, onnx 1.21.0 in a clean venv (`/home/harve/kokoro-analysis/.venv_quant`).
- No calibration set required (dynamic mode — activation quant params computed
  per inference; weights pre-quantized).
- 3 files produced in one pass; reproduction script at
  `wsl2-local:/home/harve/kokoro-analysis/quantize_tail_rest.py`.

### 2.1 Why not Conv as well

Exploratory pass with `op_types_to_quantize=["MatMul","Gemm","Conv"]` on
bucket-16 yielded rel_l2 > 0.1 in 4/12 cases (vs gate 0.05) — postnet 1D Conv
weights are bipolar with large dynamic range that QInt8 weight-only scaling
destroys. Static QDQ with calibration data might recover this, but the
incremental size win is small (Conv weights ≈ 30 % of file) so we elected to
leave it as future work — see §7.

Per-bucket A/B with 12 real misaki-phonemized cases each (6 EN + 6 ZH).
Tail-rest is fed identical pre-vocoder activations under both A (FP32) and
B (INT8) paths — isolates the INT8 delta from upstream noise.

|---|---|---|---|---|
| 8  | 12 | 0.01301 | 0.01457 | **PASS** (3.4× headroom) |
| 16 | 12 | 0.01643 | 0.01791 | **PASS** (2.8× headroom) |
| 32 | — | — | — | not measured offline; covered by live deployment + 5-shot ZH long-form bench (§5) |

Bucket-32 offline A/B skipped because radxa's bucket-32 pipeline ships

same op set** as buckets 8/16 (which pass with 2.8× headroom), and live
deployment produced 5/5 200 OK with no audio artifacts on multi-sentence ZH,
we accept the bucket-32 path on the strength of the bucket-8/16 result.

## 5. Runtime — env-gated INT8 path

Submodule `third_party/rkvoice-stream`, file
`rkvoice_stream/backends/tts/kokoro_rknn.py`. Three independent env vars,
all read at **function scope** (not module level — see
memory:trt_edge_llm_tts_env_staleness) for hot-reload safety:

```
KOKORO_RKNN_TAIL_REST_INT8_PATH              (bucket-32 path)
KOKORO_RKNN_BUCKET8_TAIL_REST_INT8_PATH      (bucket-8 router)
KOKORO_RKNN_BUCKET16_TAIL_REST_INT8_PATH     (bucket-16 router)
```

Semantics per env var: if set **and** the resolved file exists, the backend
loads INT8 in place of the corresponding FP32 tail-rest session. If unset, or
set but the file is missing, the backend logs a warning and falls back to
FP32 — fully backward-compatible.

Container env injection in `/tmp/rerun_kokoro_int8.sh` on radxa (P7a deployment
recipe, reproducible).

### 6.1 Why no speedup

ORT's CPU INT8 MatMul kernel runs at FMA throughput on ARM cores — for the
~10 MB of MatMul weights in tail-rest (vs ~16 MB of unquantized Conv weights
that dominate the tail-rest compute), the integer kernel saves cache pressure
but doesn't materially reduce wall time when the bottleneck is upstream
RKNN NPU + Conv-heavy CPU postnet. The expected 15-20 % gain assumed
MatMul-dominated runtime which doesn't hold for this network architecture.

## 7. Disposition + future work

**Decision: ship INT8 as the default code path** (env vars set, files
present). Rationale:

- Audio gate PASS with 2.8-3.4× headroom on rel_l2.
- No measurable regression.
- ~10 % smaller files → marginal RAM footprint reduction.
- Code is symmetric with bucket-8 / bucket-16 router additions, no extra
  maintenance cost.
- Trivially rollback-able: unset the 3 env vars, restart container.

**Not pursued as a perf win.** This closes the "INT8 the CPU tail" branch in
the 34 % → ??? optimization tree as **shipped but not material**. Future
perf work on the CPU tail-rest should target:

1. **Static QDQ INT8 with calibration data** including Conv ops — could
   recover the 30 % file footprint not addressed by dynamic-mode and unlock
   GFLOPs Conv kernel paths on ORT 1.26.
2. **RKNN port of the tail-rest** — would move the Conv-heavy work off CPU
   entirely. Blocked by the same `If`-rank / dynamic shape issues that block
   full vocoder-on-RKNN in the m_bucket16 build logs.
3. **Pipelined NPU + CPU** — overlap decoder-front (NPU) of frame N+1 with
   tail-rest (CPU) of frame N. Currently serial.
